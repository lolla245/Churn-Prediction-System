# Churn Prediction System
Production-ready customer churn prediction model with XGBoost, SHAP interpretability, and a real-time scoring API built with FastAPI.

## Features
- **XGBoost classifier** trained on the IBM Telco Customer Churn dataset
- **SHAP explanations** for every prediction (top contributing factors)
- **FastAPI** REST API with interactive docs at `/docs`
- Ready to deploy on free hosting (Render)

## Project structure
```
.

├── data/

├── render.yaml           # Render deployment config
└── README.md
```

## Run locally

```bash
pip install -r requirements.txt

# Train the model (generates model/ artifacts)
python train.py

# Start the API
uvicorn app.main:app --reload
```

API docs: http://localhost:8000/docs

## Example request

```bash
curl -X POST http://localhost:8000/predict \
  -H "Content-Type: application/json" \
  -d '{
    "tenure": 2,
    "Contract": "Month-to-month",
    "InternetService": "Fiber optic",
    "MonthlyCharges": 95.0,
    "TotalCharges": 190.0
  }'
```

Example response:

```json
{
  "churn_probability": 0.8376,
  "churn_prediction": "Churn",
  "top_factors": [
    {"feature": "Contract", "value": 0.0, "shap_contribution": 0.7114, "effect": "increases churn risk"},
    {"feature": "tenure", "value": 2.0, "shap_contribution": 0.5749, "effect": "increases churn risk"}
  ]
}
```

Any field not provided falls back to a sensible default, so you can send a
partial customer profile.

## Deploying to Render (free tier)

1. Push this repo to GitHub.
2. Go to https://render.com and sign in with GitHub.
3. Click **New +** → **Web Service**, select this repo.
4. Render will detect `render.yaml` automatically and configure:
   - Build command: `pip install -r requirements.txt && python train.py`
   - Start command: `uvicorn app.main:app --host 0.0.0.0 --port $PORT`
5. Click **Create Web Service**. The first deploy takes a few minutes
   (training runs during build).
6. Your live API will be available at `https://<your-service>.onrender.com`,
   with docs at `/docs`.

> Note: Render's free tier spins down on inactivity, so the first request
> after idling may take ~30-60s to respond while it wakes up.
